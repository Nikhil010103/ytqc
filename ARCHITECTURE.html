1 · What ytqc is

ytqc takes a list of YouTube channel or video IDs and, for each one, drives a real Chrome browser to extract the page, then runs a multi-stage LLM analysis to produce a structured QC record: a content category, a brand-safety verdict, audience signals, and a confidence score. Results are written to CSV/Excel.

🗣️ Conversational

You talk to it in plain language. An LLM "agent" understands intent, confirms details, and calls the right tools.

🌐 Browser-real

Extraction happens in a genuine logged-in browser via the kimi-webbridge daemon — not the API — so it sees what a human sees.

⚖️ Deterministic + LLM

Reliable rule-gates and vote aggregation wrap the LLM calls, so safety floors and categories never depend on a single model whim.

2 · Two ways to run it

Everything funnels into the same pipeline orchestrator. The only difference is what sits in front of it.

A. Interactive chat  default

Run bare ytqc → a chat REPL. You describe what you want; the LLM agent calls tools like run_qc.

# cli.py — bare invocation
ytqc            # → run_chat() REPL
› qc these channels UCxx…, UCyyy…
› 2             # confirm lanes → run_qc(...)

B. Direct CLI

Scriptable, no LLM in the loop for orchestration — straight to the pipeline.

ytqc run --input channels.csv \
     --lanes 2 --sink csv,xlsx
ytqc resume 20260614-185644-ce5eed \
     --input channels.csv
ytqc doctor   # health-check bridge + LLM
Other CLI commands: chat, version, configure (write default ~/.ytqc/config.yaml), taxonomy (show categories), accuracy (compare predictions vs gold labels).

3 · The layered architecture

Five layers, top (what the user touches) to bottom (where work happens):

Layer 1 · Interface

CLI & Chat REPL  cli.py · agent/chat.py · agent/ui.py

Typer commands; a Rich + prompt_toolkit chat UI (welcome banner, animated status line, bordered input box).

▼
Layer 2 · Agent brain

Tool-calling LLM loop  agent/loop.py · agent/tools.py · agent/schema.py · input_parse.py

An OpenAI-compatible chat model with a persona + guardrails. Tools are auto-described from Python signatures; messy pasted input is normalized into clean, deduped IDs.

▼
Layer 3 · Orchestration

Producer/consumer pipeline  pipeline/orchestrator.py · governor.py · state.py

N browser lanes (extract) + M analysis workers (LLM), decoupled by a bounded queue, governed by a rate-limiter + circuit-breaker, checkpointed for resume.

▼
Layer 4 · Extraction & Analysis

Browser bridge + LLM agents  browser/* · pipeline/*_flow.py · agents/* · llm/*

KimiClient drives Chrome to scrape pages; flows run safety-gate → vision → analyst → judge → confidence to build each QC record.

▼
Layer 5 · Output

Sinks & artifacts  sinks/* · ytqc_runs/<run_id>/

Color-coded results.xlsx, results.csv, cached extraction artifacts, and a JSONL state log.

4 · End-to-end workflow — a worked example

Following the exact input you'd paste into the chat:

› qc these channels
UCECWJfpmSWeaZ2fbb0rlq_g,channel,US - Noodah05
UC7trU46U_9XPDtMnDbiDPUQ,channel,US - JEV
UCLbdVvreihwZRL6kwuEUYsA,channel,IN - Think Music India   # …5 lines total

Step-by-step

#StageWhat happens
1Understand LLMThe agent reads your message, recognizes a QC request, and confirms the lane count (default 2) before doing anything.
2Normalize input codeparse_items() extracts only the canonical IDs — ignoring the channel / US - Name columns — and dedupes. The 5 lines become exactly 5 channel items (not 30 garbage tokens). It can also preview via inspect_input.
3Launch run coderun_qc builds the config, opens the sinks, creates a RunState (gives a run_id), and starts the Orchestrator.
4Extract browser2 browser lanes pull channels from a shared queue, drive Chrome to scrape each channel's /about + /videos, sample K representative videos, and grab their transcripts/frames/comments → a ChannelExtract bundle.
5Analyze LLM5 analysis workers consume bundles: per-video briefs → deterministic vote → channel synthesizer → optional judge → safety floor → confidence → a QCRecord.
6Persist codeEach record is written to results.csv + results.xlsx, the item is marked SUNK, and the live progress bar advances.
7Summarize LLMThe agent reads the run stats and replies in plain language: counts of unsafe / needs-review, the tier distribution, and where the file is saved.
Steps 4 & 5 run concurrently and pipelined — lane #1 can be extracting channel C while a worker analyzes channel A and lane #2 extracts channel B. That's why the throughput line reads e.g. ~215/hr | lanes: 2 workers: 5.

5 · The chat agent — brain #1

A thin multi-turn tool-calling loop around an OpenAI-compatible model (agent/loop.py). It is not the model that does QC analysis — it only orchestrates the conversation and decides which tool to call.

1System promptpersona + guardrails (confirm lanes, never invent facts, preview pastes)
2Model turnemits text and/or a tool call (JSON args)
3DispatchToolRegistry aliases + coerces args, runs the Python tool
4Tool resultfed back as a message; loop repeats until a final answer

The toolbox agent/tools.py

ToolPurpose
run_qcStart a QC run over a file path or pasted IDs/URLs/@handles (extracts + dedupes itself).
inspect_inputPreview a file or paste — counts unique channels/videos, plus what was deduped/ignored — without running.
resume_runContinue a checkpointed run from where it stopped.
show_results / list_runsFilter a run's records (e.g. only unsafe) / list past runs.
check_setup / show_taxonomyVerify the bridge + model are connected / show the category & safety vocabularies.
Tool schemas are auto-generated from each function's signature & docstring (agent/schema.py). The first docstring paragraph becomes the model-facing description — so improving a docstring improves how the agent uses the tool.

6 · The orchestrator — brain #2

A classic producer/consumer design (pipeline/orchestrator.py): browser-bound work (slow, I/O) is decoupled from LLM-bound work (CPU/network) so neither stalls the other.

             ┌────────────── shared work queue (channel/video IDs) ──────────────┐
             ▼                          ▼                                         
   LANE 0 (browser tab,      LANE 1 (browser tab,        …N lanes
   session "ytqc-lane0")      session "ytqc-lane1")      ← work-stealing
        │  extract                  │  extract
        └─────────────┬─────────────┘
                      ▼
        bounded bundle queue  (maxsize = workers + lanes)   ← back-pressure
                      │
      ┌───────────────┼───────────────┬───────────────┐
      ▼               ▼               ▼               ▼
   WORKER 0        WORKER 1        WORKER 2   …M workers   (LLM analysis)
      │  analyze → write to sinks (under a lock) → mark SUNK → advance bar
      └──────────────────────────────────────────────────────────────────

🚦 TokenBucket

Bounds the aggregate navigation rate across all lanes (max_pages_per_min, default 30) — bot hygiene. Slows adaptively under stress.

⚡ CircuitBreaker

Counts captcha/stress signals in a rolling 5-min window; retires lanes down to min_lane_count, or halts & checkpoints if degrade is off.

🛑 Halt Event

One shared flag. Ctrl-C or a fatal browser error sets it; lanes/workers observe it (even mid-backoff) and unwind cleanly.

RunStats tracked live: done, errors, unsafe, needs_review, and a tier_counts distribution — shown in the progress summary and used by the agent's recap.

7 · Browser extraction

Each lane owns a KimiClient (browser/webbridge.py) that POSTs commands to the local kimi-webbridge daemon (http://127.0.0.1:10086/command), driving a real Chrome session. Per-lane session names give true parallelism. Every navigation passes through a readiness poll, a consent-overlay click, and a captcha gate.

channel  channel_page.py

  • Navigate /about → title, subscribers, total views, country, joined date, keywords; home screenshot.
  • Navigate /videos → recent uploads grid (id, title, views, age).
  • Compute a velocity score (last-5 vs prev-5 avg views).
  • pick_sample_videos() picks K representatives: most-recent + median-age + top-viewed.
  • Each sample is then video-extracted (lite depth) and attached.

video  video_page.py

  • Navigate /watch?v=… → player response: title, author, duration, views, category, tags, description, publish date.
  • Likes via DOM aria-label.
  • Transcript (captions, sampled window) + frames (thumbnail + canvas captures).
  • Top-N comments + count (optional, default on).
  • full depth (5 frames) for direct video items; lite (2 frames) for channel samples.
Every extracted bundle is saved as artifacts/<id>/extracted.json and the item marked EXTRACTED — so a resumed run skips re-scraping.

8 · The analysis pipeline

Workers turn each extract into a QCRecord. Deterministic stages (dashed) wrap the LLM calls so safety floors and categories are never left entirely to the model.

video  run_video_flow()

·Safety pre-gateregex scan of text → rule hits + risk floors
1Vision1 image call: frames → on-screen text, kids/safety flags
2Content analyst1 merged call: tier_1/2, brand-safety, audience, keywords
3Judge?only if conflicts (~15–20%): reconcile tier/safety/language
·Post-gate + confidenceenforce floors, score, set needs_review

channel  run_channel_flow()

1Video briefsK parallel lite calls — one vote per sampled video
·Aggregateview-weighted tier vote, worst-case safety, plurality language/region
2Synthesizer1 call: channel-level record, must cite evidence to deviate from the draft
3Judge?if synth disagrees with a strong vote, or no majority
·Post-gate + confidencefloor by briefs' worst case; vote-unanimity factor

dashed = deterministic (no LLM)  ·  numbered = LLM call

Deterministic guardrails

  • Safety gate scans title/desc/tags/transcript for 8 unsafe term groups and sets minimum risk floors the LLM can't undercut.
  • Closed vocabularies (35 tier-1 categories, 4 risk levels, kids age bands…) — outputs are validated/normalized against them.
  • Confidence deducts for missing transcript/vision, tier-recovery, judge use, split votes — driving needs_review.

LLM layer llm/client.py

  • chat_json() — JSON-mode calls with <think>-stripping & truncated-JSON repair.
  • Response cache (SQLite, keyed by prompt+inputs) — repeats cost no call.
  • Concurrency semaphore (llm_concurrency, default 6) caps simultaneous model calls.
  • Temperature escalation (0.1→0.3→0.6) on parse failure; network backoff on 429/5xx.

10 · Outputs & resume

Every run lives in its own folder under output_dir (default ./ytqc_runs):

ytqc_runs/
└── 20260614-185644-ce5eed/        # run_id = YYYYMMDD-HHMMSS-<6 hex>
    ├── results.csv                  # every QCRecord, resume-safe (dedup by id)
    ├── results.xlsx                 # same data, color-coded & frozen header
    ├── state.jsonl                  # append-only EXTRACTED / SUNK checkpoints
    └── artifacts/
        └── <id>/extracted.json      # cached raw extract → skipped on resume

Excel color coding sinks/excel_sink.py

Green — safe & complete    Amber — needs review    Red — unsafe    Grey — extraction error

Resume (ytqc resume <run_id>): items already marked SUNK are skipped; items marked EXTRACTED reuse their cached artifact instead of re-scraping. State is checkpointed continuously, so an interrupted run loses no completed work.

11 · Resilience & governance

Why a run survives the messy web

  • Per-item isolation — one bad page becomes an ERROR record, never a crashed run.
  • Halt-aware backoff — retries and pauses wake instantly on Ctrl-C, so teardown is prompt and the TUI never corrupts.
  • Captcha handling — stress retires lanes or halts & checkpoints for a clean resume.
  • Logs to file — in chat mode logging is routed to ~/.ytqc/ytqc.log so background threads never scribble over the UI.

Why results stay trustworthy

  • Rule floors — deterministic gate guarantees flagged terms can't be downgraded to "safe".
  • Closed vocab validation — categories/risk levels are normalized to known values.
  • Judge step — cross-checks vision vs text vs votes and reconciles conflicts.
  • Confidence + needs_review — low-signal items are flagged for a human rather than trusted blindly.
